# SpaceSails — Detailed Implementation Plan

*Companion to [SpaceSails_plan_big_picture.md](SpaceSails_plan_big_picture.md). This document is the working plan handed to implementing agents; each milestone is a self-contained work package with acceptance criteria and a review gate.*

---

## 1. Game Concept (one paragraph)

A solar-system-scale "sailing" game. Ships are slow and heavy — they move at speeds comparable to the planets themselves — so getting anywhere is a navigation problem, not a reflex problem. The only direct control is **+10% / −10% on the current velocity vector** (plus sail/charge trim later). Like age-of-sail navigation, routes are **plotted in advance** in a map/planning mode that simulates celestial motion forward in time. The player is a **space pirate** intercepting Helium-3 cargo ships running from Saturn to the inner planets. Other ships fly hidden maneuver plans; the pirate sees only a **predicted path with growing uncertainty**, and closing an intercept is the core skill.

## 2. Design Pillars

1. **Plot, then commit.** The planning map is the primary instrument. Real-time flight is the payoff, not the main interface.
2. **Everything moves.** No static backdrop — planets, cargo ships, and plasma streams all follow trajectories. An intercept is a solution to a moving problem.
3. **Simple controls, deep consequences.** ±10% velocity increments only. Depth comes from *when*, not *what*.
4. **The in-between is the game.** Flying wide circles is easy; threading between planets (gravity, charge gradients, traffic) is where risk and reward live.
5. **Readable exotic physics.** Electric-Universe flavor mechanics (charge management, plasma streams) are deterministic and visualized, never random punishment.

## 3. Core Mechanics

### 3.1 Movement
- Ship state: position, velocity (2D, ecliptic plane — see §5 "2D decision").
- Controls: `+` scales velocity vector by 1.10, `−` by 0.90. Each pulse consumes reaction mass (budgeted per voyage). A cooldown (~1 s game-time per pulse) prevents machine-gunning.
- Later trim controls (Milestone 7): sail attitude (solar radiation pressure), charge level (electric force), thermal evaporator vent (small lateral thrust from the dark side).

### 3.2 Plotting mode (the heart of the game)
- Pause-and-plan map: time slider scrubs the future; all celestial bodies and *known* trajectories advance deterministically.
- The player drops **maneuver nodes** on their own future path: "at T+3h 12m, apply +10% ×3". The projected trajectory re-simulates instantly on every edit.
- Plans are a list of `(time, action)` pairs — exactly what AI ships also use (see 3.4), so one simulation engine serves both.
- In real-time mode the plan auto-executes; the player can override live (which invalidates the downstream plan and forces a re-plot — this is the "oh no" moment).

### 3.3 Time control
- Real-time mode runs at 1× to ~10,000× time warp (planetary transit times demand it). Warp auto-drops near encounters/hazards.
- Fixed simulation timestep regardless of warp (see §6 determinism).

### 3.4 Other ships & prediction
- Every NPC ship has a real hidden plan: waypoint schedule + maneuver list, generated by the same planner the player uses.
- The pirate's UI shows a **predicted path**: dead-reckoning from last observed position/velocity, widened into an **uncertainty cone** over time.
- Intel tightens the cone: sustained sensor lock, observed maneuvers ("she burned retrograde 20 min ago — she's braking for Mars"), captured route manifests, and *route archetypes* (cargo captains fly economical Hohmann-like transfers; the AI planner has a small set of "personalities" the player can learn).
- Guessing mechanic: player can pin a hypothesis ("assume she brakes at Mars periapsis") and the map redraws the predicted path under that assumption.

### 3.5 Piracy loop (vertical slice goal)
1. Pick a target from traffic board (departure notices from Saturn, cargo class, escort risk).
2. Plot an intercept in planning mode.
3. Fly it; adjust as the uncertainty cone collapses.
4. Endgame: match velocity within a capture threshold (relative speed < X m/s at distance < Y km) for a boarding window. Failing = flyby, re-plot or abort.
5. Fence the He3, spend on reaction mass / sails / sensors. (Economy is deliberately thin until late milestones.)

### 3.6 Electric Universe layer (Milestone 7 — flavor physics, opt-in per scenario)
- The Sun maintains a radial **charge potential field**; every ship carries hull charge that equilibrates slowly to local potential.
- Changing solar distance too fast without managing charge (venting/absorbing) builds a differential → **arcing discharge** ("Thunderbolts") that damages systems and, importantly, *lights the ship up on every sensor in the system* — stealth and charge management are the same skill.
- **Sun-side approach:** sensors looking sunward are degraded, so approaching a target from lower solar orbit is the classic pirate ambush.
- **Plasma streams:** persistent Birkeland-current "rivers" between bodies, rendered as aurora-like ribbons. A deliberately over-charged ship inside a stream gains free momentum along it (charge sign picks direction) — space's trade winds. High charge inside a stream also widens your sensor signature: fast but loud.

### 3.7 Scenarios / topographies
- **Scenario A — Sol (default):** real solar system, circular coplanar orbits at true-ish radii and periods (scaled, see §5). Ships fly Expanse/Zubrin-style Newtonian physics.
- **Scenario B — Wheel of the World (Velikovskian):** Earth–Mars–Venus aligned on a rotating spoke around Saturn (a dim proto-sun), which itself orbits the Sun. Compact, fast-moving, chaotic — the "hard mode" map.
- Scenarios are **data files** (JSON): body list, orbit parameters, charge field config, traffic routes. The engine doesn't know which cosmology it's running.

## 4. Technology Stack (Microsoft-first)

| Concern | Choice | Rationale |
|---|---|---|
| Language / runtime | **C# / .NET 10** | Installed, LTS-track, shared code client+server |
| Client app shell | **Blazor WebAssembly** (Razor components + **Bootstrap 5**) | Plain Razor+Bootstrap for all UI chrome per preference; runs in browser |
| Game rendering | **HTML5 Canvas 2D via thin JS interop** first; upgrade path to WebGL/WebGPU behind the same `IRenderer` interface | The game is vector graphics (orbits, cones, ribbons) — Canvas 2D is fast enough for hundreds of paths and keeps Milestones 1–6 simple. No heavy JS framework. |
| Simulation | **SpaceSails.Core** — pure C# class library, zero UI deps, deterministic | Same DLL runs in WASM (client prediction/plotting) and on server (authoritative) |
| Multiplayer transport | **ASP.NET Core + SignalR** (WebSockets) | MS-native, first-class Blazor support, good fit for tick-broadcast model |
| Hosting | **Azure Container Apps** (Linux container, ASP.NET Core serving the Blazor WASM statics + SignalR hub) | Scales to zero, SignalR-friendly, single container to start |
| CI/CD | **GitHub repo + GitHub Actions** → build, test, `docker build`, deploy to ACA | Owner has GitHub account; Actions has first-party Azure deploy tasks |
| Auth (later) | Microsoft Entra ID / ASP.NET Identity | Deferred to multiplayer milestone |
| Tests | xUnit + FluentAssertions | Determinism and physics regression tests are load-bearing here |

**Explicitly rejected:** Unity/Godot (not Razor/Bootstrap, heavier hosting story), Three.js/Babylon (JS-first, we stay in C#), Blazor Server for the game loop (latency + per-client server state; fine for lobby pages but the sim client must be WASM).

## 5. Simulation Design Decisions

- **2D, ecliptic plane.** Both scenarios are essentially coplanar; 2D makes plotting UI, uncertainty cones, and rendering tractable. Data structures use `Vector2d` but keep a `double Z` slot reserved so 2.5D (inclination flavor) isn't a rewrite.
- **Doubles everywhere in sim.** Solar-system coordinates in meters overflow `float` precision badly. Render layer converts to camera-relative floats (floating origin) before drawing.
- **Planets on rails.** Celestial bodies follow analytic orbits (circular to start, Keplerian later): position is a pure function of time. This is what makes plotting-mode time scrubbing instant and deterministic. The Wheel scenario is just a different analytic function (spoke rotation).
- **Ships are integrated.** Semi-implicit Euler at fixed dt (start dt = 1 s sim time; tune later), forces = gravity from rails bodies + maneuvers + (M7) charge/radiation/stream forces. Gravity is behind an `IForceField` interface so the EU charge model is a swap, not a fork.
- **Units:** SI internally (meters, seconds, kg). Scenario files may scale reality for playability (e.g., compress distances 10×, speed up orbits) — that's data, not code.
- **Trajectory projection:** re-running the same integrator forward N steps. Plot-mode responsiveness budget: full re-projection of player path (30 days sim @ dt-coarsened adaptive steps) in < 50 ms in WASM. Coarsen dt for projection, never for live sim.

## 6. Determinism & Multiplayer Model

- **Server-authoritative, client-predictive.** Server runs the one true sim at fixed tick; broadcasts ship states (and observed-event intel) at ~4 Hz per interest area; clients interpolate and run local prediction with the same Core DLL.
- Determinism rules for Core (enforce by convention + a regression test that replays a recorded 10-day scenario and asserts bit-stable positions): fixed dt, no `DateTime.Now`, no `System.Random` without injected seed, no LINQ ordering on hash sets, careful with `Math` (same runtime both sides since both are .NET — this is the big win of shared C#).
- **Hidden information lives on the server.** NPC plans and other players' maneuver nodes never ship to clients; clients only get observations (position/velocity snapshots when in sensor range) and compute uncertainty cones locally.
- Single-player = the same server sim hosted in-process (or just the WASM client running Core solo with a loopback interface). One code path, no fork.

## 8. Milestones

Each milestone ends with: tests green, a runnable demo, and a **review gate** (senior review of the diff before the next milestone starts). Sized for an implementing agent to finish without inventing architecture.

### M0 — Repo & skeleton *(review gate: structure only)*
- Create GitHub repo `spaceSails` (private), commit this plan + big-picture doc under `docs/`.
- `dotnet new` the solution per §7, .NET 10, nullable enabled, `Directory.Build.props` with analyzers.
- CI workflow: build + test on push/PR. Empty xUnit projects pass.
- **Accept:** `dotnet test` green in CI; Blazor client shows a Bootstrap page saying "SpaceSails".

### M1 — Core sim: rails + ship + ±10% *(the physics seed — review carefully)*
- `ICelestialEphemeris` with circular orbits from `sol.json` (Sun + 8 planets, scaled radii/periods defined in the scenario file).
- `Simulator` fixed-dt integration of one ship under point-mass gravity of all bodies; `ManeuverPlan` execution (±10% pulses at scheduled times).
- `Simulator.Project(state, plan, horizon)` returning a polyline.
- Determinism replay test + basic orbital sanity tests (circular orbit stays circular within tolerance over 1 sim-year).
- **Accept:** console/test harness prints a ship falling sunward and a plan that circularizes it.

### M2 — Renderer & camera
- Canvas interop layer (`renderer.js` + C# `CanvasRenderer : IRenderer`): draw circles, polylines, text, screen-space icons; requestAnimationFrame loop calling into .NET.
- Camera: pan, zoom (mouse wheel, ~12 orders of magnitude — whole system → ship close-up), floating origin, follow-ship mode.
- Render scenario A live at time warp with a warp control (Bootstrap toolbar: 1×…10,000×, pause).
- **Accept:** browser shows planets orbiting; smooth at 60 fps with 500 dummy trajectory polylines.

### M3 — Fly the ship (real-time single player)
- Player ship in the loop: `+`/`−` keys apply velocity pulses (with cooldown + reaction-mass budget), HUD (Bootstrap overlay: velocity, mass remaining, warp, nearest body).
- Warp auto-drop near bodies. Trajectory ribbon ahead of the ship (Project on every pulse).
- **Accept:** you can leave Earth, do a (ugly) Mars flyby by hand, and it *feels controllable* — this validates the tested mechanic from the big-picture doc.

### M4 — Plotting mode
- Pause-map: time scrubber, future positions of all bodies ghosted, player path projected.
- Maneuver nodes: add/move/delete on own path; each node = burst of ± pulses; instant re-projection; plan persists and auto-executes back in real-time mode.
- Live override invalidates downstream nodes (visually: they turn red/dashed).
- **Accept:** plot Earth→Mars intercept entirely in map mode, press play, arrive within capture threshold without touching controls.

### M5 — Traffic & prediction
- NPC planner in Server host: generates He3 cargo routes Saturn→inner planets as `ManeuverPlan`s (2–3 route personalities: economical, fast, evasive).
- `IObservationModel`: sensor range, sun-glare direction penalty; observations logged with timestamps.
- `PathPredictor`: dead-reckon + uncertainty cone (grows with time-since-observation and target's remaining plausible maneuvers); hypothesis pinning ("brakes at Mars").
- Traffic board UI (Bootstrap table): departures, cargo class, last-seen.
- **Accept:** cone visibly tightens as you shadow a target; a pinned correct hypothesis makes the predicted and actual paths converge.

### M6 — Piracy vertical slice 🏴‍☠️
- Capture rule (rel-velocity + distance window for N seconds), boarding resolution (simple first: automatic success, cargo transferred), fence/upgrade screen (reaction mass, sensor range, cargo hold).
- Failure states: out of reaction mass (drift — rescue fee = lose cargo), missed windows.
- Single scripted "first hunt" tutorial voyage.
- **Accept:** full loop — pick target, plot, chase, capture, sell, upgrade — is playable and fun-ish. **This is the go/no-go review for everything after.**

### M7 — Electric Universe layer
- Charge potential field + ship charge state, equilibration, venting control, arcing (damage + system-wide sensor ping), sun-side sensor shadow, plasma streams as scenario-defined ribbon paths with along-stream force ∝ charge.
- All under `IForceField`/scenario flags so Scenario A can run pure-Newtonian.
- **Accept:** the low-solar-orbit sneak-up ambush works; riding a stream Saturn→Jupiter is faster but gets you spotted.

### M8 — Wheel of the World scenario
- `wheel.json`: Saturn as sub-center with Earth–Mars–Venus on the rotating spoke; tune traffic routes for it.
- **Accept:** both scenarios selectable at start; Wheel is playably harder.

### M9 — Multiplayer
- SignalR hub: join session, authoritative tick, interest-managed state broadcast, command channel (maneuver nodes + live pulses), server-side observation filtering (hidden info stays hidden).
- Client reconciliation (snap + re-project). 2–8 players, pirates in a shared traffic system (PvE first; PvP piracy = stretch).
- **Accept:** two browsers hunt in one system; a ship maneuvering outside your sensor range does *not* update on your map.

### M10 — Azure deployment & polish
- Dockerfile, `deploy.yml` → Azure Container Apps; app config via ACA env vars/secrets; basic telemetry (Application Insights).
- Polish pass: audio cues, arc/aurora visual effects, onboarding text.
- **Accept:** public URL, friend joins from their machine, a hunt completes.

## 9. Working Agreement for Implementing Agents

- **Never widen a milestone.** If a task needs an interface that doesn't exist, add the minimal version in Core and note it in the PR description.
- **Determinism is law** in `SpaceSails.Core`: no wall-clock, no unseeded random, no environment-dependent math. The replay test must stay green.
- **UI = Razor + Bootstrap only.** Custom CSS is allowed for the canvas layer/HUD; no other CSS or JS frameworks. JS lives only in `renderer.js` (+ future `input.js`).
- One PR per milestone task, small commits, tests in the same PR as the code they cover.
- ADRs (`docs/adr/NNN-title.md`) for any deviation from this plan; the plan file is updated in the same PR.

## 10. Open Questions (decide at the relevant milestone, not before)

1. **Scale factor** for Scenario A — real distances with aggressive warp, or 10× compressed? Decide in M3 playtesting (feel > realism).
2. Boarding minigame vs. auto-resolve long-term (M6 review).
3. Escorts / combat — out of scope until the piracy loop is proven fun (post-M6).
4. Persistence (voyage save / career) — likely a small `System.Text.Json` save file client-side pre-multiplayer, server-side (Azure Table/Cosmos) after M9.
5. WebGL upgrade trigger: if M5 profiling shows Canvas 2D < 60 fps with full traffic + cones + ribbons.
